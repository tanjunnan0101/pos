# Preserve sidebar nav scroll on staff route change

## GitHub Issues
- **Issue:** https://github.com/satisfecho/pos/issues/215
- **215**

## Problem / goal
Staff pages embed `<app-sidebar>` inside each route component (e.g. orders, products, inventory). Navigating between staff routes destroys and recreates the sidebar, so the nav list scroll position resets to the top. Users with many nav items lose their place and must scroll again on every route change.

Short-term fix: on `NavigationEnd`, scroll the active nav link into view and optionally persist/restore the nav container `scrollTop` across staff route changes. Long-term (optional follow-up, out of scope unless trivial): a single staff shell layout with one persistent `app-sidebar` instance.

## High-level instructions for coder
- Inspect `front/src/app/shared/sidebar.component.ts` and `.scss` — nav lives in `<nav class="nav">` inside `<aside class="sidebar">`; many staff routes wrap content with `<app-sidebar>…</app-sidebar>`.
- Subscribe to router `NavigationEnd` (or use existing router hooks in sidebar or `StaffLayoutService` in `front/src/app/services/staff-layout.service.ts`) to call `scrollIntoView` on the active `.nav-link` (respect collapsed nav state if applicable).
- Optionally persist nav `scrollTop` in `sessionStorage` or `StaffLayoutService` keyed by tenant/user so it survives route recreation; restore on init after nav links render.
- Verify on desktop and mobile: navigate between distant nav items (e.g. Settings ↔ Inventory) — scroll position or active link visibility should not jump to top unexpectedly.
- Do not change unrelated layout, permissions, or i18n unless needed for accessibility labels on scroll behavior.
- Smoke: confirm `docker logs pos-front` shows successful bundle after changes; manually test staff navigation in browser.

## Implementation notes
- `StaffLayoutService`: `getNavScrollTop` / `setNavScrollTop` persist scroll in `sessionStorage` keyed by tenant/user (`t{tenant_id}-u{user_id}`).
- `SidebarComponent`: `#navScroll` template ref; saves scroll on `(scroll)` and `ngOnDestroy`; restores on init/`NavigationEnd`; `scrollIntoView({ block: 'nearest' })` on active `.nav-link` / `.nav-sublink` when nav is not collapsed.
- **Mobile fix (2026-05-28):** `closeSidebar()` calls `persistNavScroll()` before closing the drawer.
- **Remaining (2026-05-28 retest):** `ngOnDestroy()` still calls `persistNavScroll()` with `scrollTop` **0** after route change and overwrites the value saved in `closeSidebar()` — criterion **(5)** still **FAIL**.
